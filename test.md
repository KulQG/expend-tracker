develop / собственная ветка с фичей/фиксом (feat/...)

git revert <hash> / git reset

## 2. JavaScript

### 2.1. Что такое Event Loop?

Механизм управляющий асинхронными операциями. Синхронный код -> microtasks (Promise) -> Macrotasks (setTimeout, setInterval, fetch, XMLHttpRequest)

---

### 2.2. Каким образом можно дождаться исполнения нескольких асинхронных операций, не блокируя тред/поток?

const [a,b] = await Promise.all([...])

---

## 3. TypeScript

### 3.1. Чем отличается `any` от `unknown`?

any используется для обозначения вообщего любого типа, убивает типизацию
unknown используется для обозначения типа, который заранее неизвестен, но перед использованием переменной с этим типом, требует присваивания конкретного типа

---

### 3.2. Что такое function overloading (перегрузка функций)?

Позволяет сделать несколько реализаций одной функции, которые будут определяться автоматически в зависимости от переданных аргументов

---

## 4. Node.js

### 4.1. Поддерживает ли Node.js Event Loop?

Да, но отличается от браузерного

---

### 4.2. При работе со стандартной библиотекой Node.js, как использовать современные функции, работающие на Promise, а не callback'ах?

импорт import fs from 'node:fs/promises' вместе с await 

---

## 5. React.js

### 5.1. Как совершить единовременный HTTP-запрос при рендеринге компонента?

Выполнить запрос в хуке useEffect с пустым массивом зависимостей

---

### 5.2. Что такое key (ключ), и зачем он нужен?

проп для идентификации конкретных элементов списка. Нужен для оптимизации рендеринга (чтобы при манипуляциях со списком, он не рендерился заново весь)

---

## 6. Postgres

### 6.1. Чем отличаются `LEFT JOIN` и `INNER JOIN`?

LEFT JOIN вернет всю левую таблицу + совпадения из правой, INNER JOIN вернет только совпадающие строки

---

### 6.2. Что такое индекс? Зачем нужно индексировать колонки/поля в таблицах?

Структура в БД которая позволяет быстро находить записи в таблице. Используется для ускорения поиска в больших таблицах.

---

## 7. Отношение к ИИ

### 7.1. Пользуетесь ли ИИ/LLM, если да, то какими сервисами?

Да, Claude, Gemini, Perplexity для быстрого поиска

---

### 7.2. Каким образом используете ИИ/LLM? Каковы риски? (или: почему не пользуетесь?)

- быстро получить документацию к какой то либе, сделать ревью кода, объяснить баг, отдать простые задачи для ускорения разработки, показать пример и реализацию
- Самый большая проблема LLM, потеря контекста + даже Claude код пишет не всегда по правилам SOLID, KISS, DRY, YAGNI просто потому что учился на коде других людей, которые не всегда пишут качественно. Совершает большое количество ошибок из-за актуальности библиотек, не всегда понимает как актуально использовать что-то. Много проблем и всегда нужно перепроверять.

---

### 7.3. Как бороться с перечисленными вами рисками?

Писать очень сильные промпты с огромной детализацией, перепроверять код, либо вообще не использовать LLM. Бывает, что быстрее самому написать логику, чем пытаться объяснить что-то нейросети :). А ей отдать на ревью

---

# Тестовое задание

Создайте **offline-first PWA трекер расходов** на TypeScript, используя React и Node.js.

Исходный код должен быть Git-репозиторием. Варианты сдачи:

1. Публичный хостинг репозиториев (GitHub / GitLab / Codeberg / etc.)
2. Сжатый архив (предпочтительно `tar.gz`)

---

## 1. Backend

Бэкенд на Node.js + Express.js.

### 1.1. База данных

- Подключение к PostgreSQL
- Конфигурация через environment variables

### 1.2. Таблица статей расходов

Используя любой коннектор/ORM, создать таблицу со следующими полями:

| Поле   | Описание                  |
|--------|---------------------------|
| `id`   | Уникальный идентификатор  |
| `name` | Название статьи           |
| `sum`  | Сумма статьи              |
| `date` | Дата статьи               |

> Тип данных, дополнительные поля и прочее - на усмотрение разработчика.

---

## 2. Frontend

React + Vite.

### 2.1. Хедер

Статус подключения к сети / бэкенду с живым апдейтом.

### 2.2. Форма создания статьи расходов

| Поле            | Валидация                        |
|-----------------|----------------------------------|
| Название статьи | 3 ≤ x ≤ 30 символов              |
| Сумма           | 1.00 ≤ x ≤ 1 000 000.00 у.е.    |
| Дата            | Любая                            |

### 2.3. Список статей

Рядом с формой - список созданных статей с итоговой суммой расходов. Бонусом будет:
- Корректный расчет суммы у.е.
- Имплементация удаления уже созданных статей
- Функционал редактирования данных уже созданных статей делать не обязательно

### 2.4. Offline-first

При отсутствии подключения созданные статьи сохраняются локально. После восстановления соединения - автоматическая синхронизация с бэкендом.

> Стиль, библиотеки компонентов и прочее - на усмотрение разработчика.
